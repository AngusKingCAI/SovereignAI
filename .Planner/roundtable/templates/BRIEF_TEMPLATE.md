# 12-Section Brief Template

**Version**: 1.0  
**Last Updated**: 2026-07-22  
**Status**: Active

## Purpose
Provide comprehensive 12-section brief template for Round Table review following BP-based best practices.

## BP-Based Brief Design Principles

### Core Principles (Per Research)

1. **Comprehensive Context**: Provide all necessary context for panelists to make informed evaluations
2. **Structured Format**: Use consistent 12-section structure for predictability and efficiency
3. **Evidence-Based**: Include specific evidence and data to support evaluation
4. **Panelist-Specific**: Tailor information to panelist competencies and assignments
5. **Brief Efficiency**: Balance comprehensiveness with brevity to respect panelist time

## Brief Template Structure

### Section 1: Context

**Purpose**: Provide project background and batch objectives

**Content Requirements**:
- Project background and context
- Batch objectives and goals
- Historical context (previous related work)
- Current state and why this batch is needed
- Success criteria for the batch

**Template**:
```markdown
## Context

{Project background and context}

{Batch objectives and goals}

{Historical context - previous related work}

{Current state and why this batch is needed}

{Success criteria for the batch}
```

**Example**:
```markdown
## Context

SovereignAI workflow optimization batch targeting hard gating infrastructure. Plan 34 experienced critical failure: committed and tagged despite no attestation, incomplete traces, and verification run on wrong plan (Plan 33). Root cause: lack of external enforcement layer. This batch implements immutable plan identity, fail-closed defaults, Devin Local SWE 1.6 hooks, and trace enhancement.

Success criteria: All plans in batch must achieve ≥90% Round Table score with no CRITICAL findings.
```

### Section 2: Plans in Batch

**Purpose**: List all plans in batch with dependencies and metadata

**Content Requirements**:
- Complete list of plans in batch
- Plan titles and numbers
- Dependency relationships between plans
- Vision principles assigned to each plan
- AR (Architecture Rules) and OR (Operational Rules) assigned to each plan

**Template**:
```markdown
## Plans in this batch

| Plan | Title | Depends On | Vision Principles | AR Rules | OR Rules |
|------|-------|------------|-------------------|----------|----------|
| {Plan Number} | {Plan Title} | {Dependencies} | {Vision Principles} | {AR Rules} | {OR Rules} |
```

**Example**:
```markdown
## Plans in this batch

| Plan | Title | Depends On | Vision Principles | AR Rules | OR Rules |
|------|-------|------------|-------------------|----------|----------|
| 35 | Workflow Optimization (Foundation) | 34 | P1, P2, P9, P13 | AR1, AR2, AR8, AR10, AR11, AR12 | UOR-1, UOR-2, VOR-1, COR-2 |
| 35.1 | Devin Configuration & Hooks | 35 | P2, P9, P13 | AR1, AR8 | UOR-1, UOR-2 |
```

### Section 3: Rules Carried Forward

**Purpose**: List all governance rules applicable to this batch

**Content Requirements**:
- All AR (Architecture Rules) that apply
- All OR (Operational Rules) that apply
- Any specific rule interpretations for this batch
- Rule conflicts or ambiguities (if any)

**Template**:
```markdown
## Rules carried forward

{AR rules range and key rules}
{OR rules range and key rules}
{Any specific interpretations or notes}
```

**Example**:
```markdown
## Rules carried forward

AR1-AR18 (ARCHITECTURE.md), UOR-1 to SOR-2 (OR_RULES.md), M1-M8 (LANDMINES.md)

Key rule interpretations:
- AR11: Hard gate enforcement takes priority over workflow efficiency in this batch
- UOR-1: Token tracking limitations apply given Devin's closed system
```

### Section 4: Questions for Round Table

**Purpose**: Provide specific questions for panelists to address

**Content Requirements**:
- 3-5 targeted questions for panelists
- Questions should be specific and answerable
- Questions should address key risks or uncertainties
- Questions should be mapped to relevant competencies

**Template**:
```markdown
## Questions for Round Table

Q1: {Question text} (Competency: {Competency Name})
Q2: {Question text} (Competency: {Competency Name})
Q3: {Question text} (Competency: {Competency Name})
Q4: {Question text} (Competency: {Competency Name})
Q5: {Question text} (Competency: {Competency Name})
```

**Example**:
```markdown
## Questions for Round Table

Q1: Is Devin hooks.v1.json the correct approach for hard gating, or should we use git hooks instead? (Competency: Technical Architecture)
Q2: Should plan lock file be human-created or auto-generated by Executor at /open? (Competency: Technical Architecture)
Q3: What is the optimal balance between automated enforcement and human confirmation gates? (Competency: Cross-Team Impact)
Q4: How should token usage tracking be implemented given Devin's closed system? (Competency: Technical Architecture)
Q5: What is the maximum acceptable plan cycle time increase from adding hard gates? (Competency: Cross-Team Impact)
```

```markdown

{DD numbers and descriptions of resolved questions}
```

**Purpose**: Document known risks and mitigation strategies

**Content Requirements**:
- List of identified risks
- Risk severity (CRITICAL, HIGH, MEDIUM, LOW)
- Mitigation strategies for each risk
- Landmine pre-screen results

**Template**:
```markdown

R1: {Risk description} (Severity: {Severity})
R2: {Risk description} (Severity: {Severity})
R3: {Risk description} (Severity: {Severity})

Landmine pre-screen: {No blocking landmines triggered / Specific landmines triggered}
```

**Example**:
```markdown
## Risks flagged

R1: Devin hooks.v1.json may not support all claimed features in SWE 1.6 (Severity: HIGH)
R2: Plan lock file adds friction to every plan execution (Severity: MEDIUM)
R3: Human confirmation gates may slow workflow significantly (Severity: MEDIUM)
R4: Parallel check execution may introduce race conditions (Severity: MEDIUM)
R5: Incremental checking may miss cross-file dependencies (Severity: LOW)

Landmine pre-screen: No blocking landmines triggered
```

```markdown
- Web search integration for evidence gathering

- Rubric-based scoring with behavioral indicators
```

```markdown
- Competency-specific evaluation per panelist assignment
```

## Brief Assembly Guidelines

### Assembly Process

1. **Context First**: Start with comprehensive context to frame the batch
2. **Plan Structure**: Clearly define plan relationships and dependencies
3. **Rule Framework**: Establish applicable rules and interpretations
4. **Targeted Questions**: Formulate specific, answerable questions
5. **Risk Assessment**: Document known risks and mitigations
6. **Panelist Focus**: Tailor profiles and prompts to competency assignments
7. **Rubric Integration**: Ensure rubrics align with competency definitions

### Quality Checks

- **Completeness**: All 12 sections present and complete
- **Consistency**: Information consistent across sections
- **Clarity**: Language clear and unambiguous
- **Relevance**: All information relevant to evaluation
- **Accuracy**: All information accurate and up-to-date

### Brief Length Guidelines

- **Total Length**: 1000-2000 words (approximately 2-4 pages)
- **Section Balance**: Each section proportional to importance
- **Density**: High information density, no fluff
- **Scannability**: Use tables, bullets, and formatting for quick scanning

## Compliance

Post compliance line after brief assembly:
`✅ Gate BRIEF-ASSEMBLY PASS: 12-section brief assembled with {N} sections, {word_count} words following BP principles`

## Evolution Condition

Brief template evolves when new sections are needed, BP research updates brief structure, or feedback indicates need for improvement.
